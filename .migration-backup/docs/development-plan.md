# 개발 계획

## 목적

이 문서는 현재 저장소에서 먼저 만들어야 할 동작 가능한 최소 제품(MVP)의 개발 계획을 설명한다.

목표는 아키텍처 경계를 완벽하게 분리하는 것이 아니라, 사용자가 웹 UI에서 종목을 입력하고, 백엔드 워크플로우가 판단 결과와 주문 계획을 만들고, 그 결과를 저장한 뒤 다시 조회할 수 있는 전체 제품 흐름을 완성하는 것이다.

투자 전략, 실제 외부 데이터 API, 실브로커 주문, 고급 분석 지표는 이 문서의 현재 범위가 아니다. 해당 항목은 MVP 이후 확장 단계에서 정한다.

## MVP 목표

MVP는 아래 흐름이 끝까지 동작하는 것을 목표로 한다.

```text
사용자
-> 웹 UI에서 종목 입력
-> 백엔드 API 호출
-> MainAgent 워크플로우 실행
-> 데이터 수집 stub
-> 분석 stub
-> 리스크 검증 stub
-> 매수/매도 판단
-> 주문 계획 생성
-> 평가 로그 생성
-> 결과 저장
-> 웹 UI에서 결과와 히스토리 확인
```

현재 MVP에서는 실제 외부 API와 실주문을 호출하지 않는다.

## 핵심 설계 원칙

* 먼저 끝까지 동작하는 제품 흐름을 만든다.
* 실제 데이터, 실제 브로커, 실제 저장소 확장은 MVP 이후 단계로 둔다.
* `MainAgent`는 현재 워크플로우 순서를 유지한다.
* 주문 계획은 생성하지만 실제 주문 제출은 하지 않는다.
* 결과 저장과 조회는 MVP 제품 기능으로 포함한다.
* 구조 분리는 제품 흐름을 방해하지 않는 최소 범위에서 진행한다.
* 미확정 투자 전략과 브로커 세부 계약은 `TBD`로 남긴다.

## 1주차: 현재 앱을 실제 사용 가능한 흐름으로 정리

목표는 웹에서 종목을 입력하면 백엔드 판단 결과가 화면에 나오는 것이다.

작업:

* FastAPI `/decisions` 응답 구조 정리
* 프론트엔드에서 심볼 입력 폼 구현
* 프론트엔드에서 `/decisions` 호출
* decision, order plan, risk result를 화면에 표시
* 에러 상태와 로딩 상태 추가
* 현재 stub workflow는 유지

산출물:

* 사용자가 `AAPL, MSFT` 같은 심볼을 입력할 수 있다.
* 결과 화면에서 판단과 주문 계획을 확인할 수 있다.
* 백엔드와 프론트엔드가 연결된다.

## 2주차: 결과 저장 기능 추가

목표는 한 번 생성한 판단 결과를 다시 볼 수 있게 하는 것이다.

작업:

* `repositories/` 패키지 추가
* `DecisionLogRepository` interface 정의
* 초기 저장 구현 추가
* decision 생성 시 결과 저장
* `GET /decisions` 목록 API 추가
* `GET /decisions/{id}` 상세 API 추가

초기 저장 구현은 SQLite를 우선 검토한다. 단, MVP 속도가 더 중요하면 memory repository로 시작하고 SQLite는 다음 작업으로 넘긴다.

산출물:

* 판단 기록이 저장된다.
* 과거 판단 목록을 조회할 수 있다.
* 특정 판단 상세를 조회할 수 있다.

## 3주차: 웹 UI에 히스토리 화면 추가

목표는 사용자가 이전 판단들을 볼 수 있는 최소 운영 화면을 만드는 것이다.

작업:

* 최근 decision 목록 표시
* decision 상세 보기 구현
* 새 판단 생성 화면과 상세 화면 연결
* 결과 화면에 요청 심볼, 분석 점수, 리스크 승인 여부 표시
* 결과 화면에 매수/매도/HOLD 판단 표시
* 결과 화면에 주문 제출 가능 여부와 평가 요약 표시

산출물:

* 단발성 API demo가 아니라 제품처럼 사용할 수 있다.
* 새 판단 생성과 과거 판단 조회가 같은 UI 안에서 가능하다.

## 4주차: 아키텍처 경계 최소 분리

목표는 제품 흐름을 유지하면서 나중에 실제 데이터와 브로커를 붙일 수 있는 최소 경계를 만드는 것이다.

작업:

* `providers/` 패키지 추가
* `DataCollectionAgent`의 stub 데이터 생성 로직을 provider 경계로 이동
* `brokers/` 패키지 추가
* `OrderExecutionAgent`가 실제 주문 없이 주문 계획만 생성하는 계약 유지
* 브로커 adapter 추가를 위한 기본 경계 준비
* `api/dependencies.py`를 추가해 `MainAgent` 조립 로직 분리

산출물:

* 전체 제품 동작은 유지된다.
* 데이터 provider 교체가 가능해진다.
* 브로커 adapter 추가를 위한 자리가 생긴다.
* API 진입점과 조립 코드가 분리된다.

## 5주차: 구조 테스트와 API 테스트 보강

목표는 MVP 흐름이 깨지지 않게 최소 회귀 테스트를 만드는 것이다.

작업:

* `POST /decisions` 테스트 추가
* `GET /decisions` 테스트 추가
* `GET /decisions/{id}` 테스트 추가
* `MainAgent` workflow 테스트 유지
* 리스크 미승인 시 주문 제출 불가 테스트 유지
* repository 저장과 조회 테스트 추가

산출물:

* 핵심 제품 흐름 테스트가 생긴다.
* API 계약 테스트가 생긴다.
* 저장소 테스트가 생긴다.

## 우선순위

MVP의 작업 우선순위는 아래 순서로 본다.

```text
1. 웹에서 판단 생성
2. 백엔드 workflow 결과 반환
3. 결과 저장
4. 히스토리 조회
5. 최소 구조 분리
6. 테스트
7. 문서
8. 품질 검증
```

## 수정 트리거

아래 상황이 생기면 이 문서를 갱신한다.

* MVP 범위가 바뀜
* API endpoint 계약이 바뀜
* UI의 핵심 제품 흐름이 바뀜
* 저장 방식이 바뀜
* 실제 데이터, 브로커, 백테스트가 MVP 범위로 들어옴
* 주차별 작업 순서나 검증 기준이 바뀜

## 현재 요청 기준 프론트 재구성 플랜

### 목표 화면 구조

요청 기준의 목표 구조는 아래 5개 화면이다.

* `대시보드`: 메인 에이전트와 채팅하는 단일 허브
* `에이전트`: 메인 에이전트와 각 서브 에이전트의 프롬프트 및 특화 설정 관리
* `히스토리 타임라인`: 메인 에이전트와 서브 에이전트의 실행 순서와 상태 표시
* `보고서`: 날짜순 보고서 리스트와 상세 보기
* `계좌`: 계좌 연결과 연결된 계좌 상태 확인

### 현재 구현과 다른 점

* 현재 대시보드는 채팅 전용이 아니라 실행 폼, 최근 런, 에이전트 카드, 저장 패널을 함께 둔다.
* 현재 에이전트 화면은 메인 에이전트 전용 설정 진입점과 에이전트별 특화 설정 저장 계층이 없다.
* 현재 히스토리 타임라인은 전용 API와 전용 페이지 계약이 없다.
* 현재 보고서는 저장 실행 복기가 가능하지만, 타임라인과 메인 에이전트 흐름과는 분리되어 있다.
* 현재 계좌 연결은 브라우저 로컬 입력 위주이며 Toss API, 한국투자 API 연결 정보를 서버 계약으로 저장하거나 검증하지 않는다.

### 단계별 구현 순서

#### 1단계: 정보 구조와 네비게이션 정리

목표는 화면 구조를 요청한 5개 영역으로 단순화하는 것이다.

작업:
* 사이드바 메뉴를 `대시보드`, `에이전트`, `히스토리`, `보고서`, `계좌` 중심으로 재구성
* `프롬프트 설정` 화면을 독립 메뉴에서 제거하고 `에이전트` 화면으로 통합
* 대시보드의 카드형 홈 구조를 메인 에이전트 채팅 중심 구조로 교체
* 메인 에이전트 진입점을 `에이전트` 화면 최상단 그룹으로 승격

산출물:
* 사용자가 진입 즉시 5개 핵심 화면만 인지한다.
* 메인 에이전트 채팅과 에이전트 설정이 서로 다른 목적의 화면으로 분리된다.

#### 2단계: 대시보드 채팅 UX 재구성

목표는 메인 화면을 ChatGPT형 워크플로우 콘솔로 바꾸는 것이다.

작업:
* 새 run 생성과 기존 run follow-up을 하나의 채팅 컴포저로 통합
* `POST /decisions` 호출을 첫 메시지 실행 흐름으로 연결
* `POST /console/interactions` 호출을 후속 질문과 재실행 흐름으로 연결
* 채팅 응답 카드에 `summary`, `watch_symbols`, `decisions`, `suggested_actions`를 집중 노출
* run context rail에 `run_id`, runtime, 현재 목적, 보고서 저장 액션만 유지

산출물:
* 대시보드는 메인 에이전트와 대화하는 단일 허브가 된다.
* 기존의 분산된 입력 패널이 하나의 메시지 중심 흐름으로 합쳐진다.

#### 3단계: 에이전트 설정 화면 확장

목표는 메인 에이전트와 서브 에이전트 모두를 설정 가능한 관리 허브로 만드는 것이다.

작업:
* `main_agent`를 포함하도록 프롬프트 목록 계약과 프론트 타입 확장
* 에이전트별 공통 항목과 특화 항목을 구분한 설정 스키마 설계
* 에이전트 상세 화면에 `prompt editor`, `settings form`, `latest output` 세 영역 배치
* 메인 에이전트 전용 화면에 최근 응답 흐름 요약과 목적 설정 표시

백엔드 선행 작업:
* `main_agent`가 `GET /agent-prompts` 응답에 포함되도록 정리
* 에이전트별 설정 저장/조회 API 설계

산출물:
* 메인 에이전트와 각 서브 에이전트의 설정 책임이 한 화면군에서 관리된다.
* 프롬프트 수정과 설정 수정을 분리하지 않고 같은 맥락에서 처리할 수 있다.

#### 4단계: 히스토리 타임라인 계약 추가

목표는 메인 에이전트와 서브 에이전트의 작업 흐름을 시간축으로 보여주는 것이다.

작업:
* run 목록 API와 run 상세 API 추가
* run 상세에 `agent_statuses`, `timeline`, `analysis_summaries`, `result` 포함
* 재실행 분기와 follow-up 메시지 반영 여부를 이벤트로 기록
* 프론트에서 좌측 run 목록, 우측 타임라인 상세 패널 구성

백엔드 선행 작업:
* 현재 `history` 모델을 실제 라우트와 서비스에 연결
* in-memory run 저장소를 목록/상세 조회 가능한 형태로 확장

산출물:
* 사용자가 메인 에이전트와 서브 에이전트의 작업 순서를 한눈에 본다.
* 보고서와 별도로 실행 과정 자체를 복기할 수 있다.

#### 5단계: 보고서 화면 정리

목표는 저장된 보고서를 날짜순으로 빠르게 찾고 상세 내용을 읽게 하는 것이다.

작업:
* `GET /experiments` 결과를 날짜순 정렬 기준으로 고정
* 목록에는 이름, 생성시각, 심볼, 최종 액션만 compact하게 노출
* 상세에는 메인 요약, 보고서, 판단, 주문 계획, 수집 데이터 순으로 배치
* 히스토리 화면에서 선택한 run을 보고서 저장으로 넘기는 흐름 연결

산출물:
* 보고서 화면은 저장 결과 확인과 복기에 집중한다.
* 대시보드와 히스토리에서 생성된 결과가 보고서 화면으로 자연스럽게 연결된다.

#### 6단계: 계좌 연결과 상태 조회 확장

목표는 사용자가 자신의 계좌를 연결하고 일반 투자앱 수준의 기본 상태를 보는 것이다.

작업:
* 계좌 연결 폼을 `broker kind`, `account identifiers`, `credential status` 중심으로 재구성
* Toss API 입력란과 한국투자 API 입력란을 브로커 종류에 따라 분기
* `GET /account` 응답에 연결 상태와 자격 증명 상태를 일관되게 포함
* 필요 시 계좌 연결 저장/조회 API 추가

백엔드 선행 작업:
* `AccountResponse`의 `connection`, `credential_status` 계약을 실제 구현과 맞춤
* Toss 연동 범위를 별도 adapter 경계로 설계
* 한국투자 계좌와 Toss 계좌를 구분할 broker adapter 선택 계약 정의

산출물:
* 사용자는 어떤 계좌가 연결되었는지와 현재 계좌 상태를 같은 맥락에서 본다.
* 계좌 화면이 로컬 메모 입력 화면이 아니라 실제 연결 화면으로 바뀐다.

#### 7단계: 검증과 문서 동기화

목표는 새 구조와 공개 계약을 함께 고정하는 것이다.

작업:
* API 계약 문서 갱신
* 프론트 사용 안내 문서 갱신
* 히스토리와 계좌 관련 테스트 추가
* `UV_CACHE_DIR=/tmp/uv-cache uv run pytest` 실행
* 구현 후 `UV_CACHE_DIR=/tmp/uv-cache uv run ruff format .` 실행
* 구현 후 `UV_CACHE_DIR=/tmp/uv-cache uv run ruff check .` 실행

산출물:
* 문서와 코드가 같은 화면 구조를 가리킨다.
* 새 정보 구조가 테스트로 고정된다.

### 구현 우선순위

```text
1. 대시보드 채팅 단순화
2. 메인 에이전트를 포함한 에이전트 설정 구조
3. 히스토리 타임라인 backend contract
4. 보고서 리스트/상세 정리
5. 계좌 연결 계약 확장
6. 테스트와 문서 동기화
```
