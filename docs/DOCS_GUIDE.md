# DOCS GUIDE

## 목적

`docs/`는 이 저장소의 현재 구현 계약, 구조, 운영 규칙을 유지하는 공식 문서 공간이다.

테스트 작성, 유지, 리뷰 기준은 `TESTS_GUIDE.md`에서 별도로 관리한다.

## 현재 루트 문서 구성

현재 `docs/` 루트에는 아래 문서를 유지한다.

* `README.md`
* `DOCS_GUIDE.md`
* `TESTS_GUIDE.md`
* `project-overview.md`
* `architecture.md`
* `repository-map.md`
* `development-plan.md`
* `agent-structure.md`
* `domain-rules.md`

현재 코드가 없는 가상의 서브시스템 문서나 다른 프로젝트용 템플릿 문서는 추가하지 않는다.

## 구조 규칙

1. 문서는 현재 저장소의 실제 코드 구조와 책임 경계를 기준으로 나눈다.
2. 문서 전용 분류를 만들더라도 현재 코드나 운영 계약과 연결되지 않으면 두지 않는다.
3. `docs/` 및 하위 디렉터리가 생기면 각 디렉터리에 `README.md`를 둔다.
4. 각 `README.md`에는 하위 문서 역할, 관련 코드 경로, 수정 트리거를 적는다.

## 문서 단위 규칙

1. 한 문서는 한 주제 또는 한 계약 집합을 우선 설명한다.
2. 프로젝트 전반 개요는 `README.md`와 `project-overview.md`에 두고, 세부 책임은 개별 문서로 분리한다.
3. 구조, 흐름, 책임, 운영 기준을 우선 문서화하고, 자잘한 구현 설명은 코드 주석으로 남긴다.
4. 현재 구현에 없는 미래 설계나 장기 아이디어는 공식 계약 문서처럼 쓰지 않는다.
5. 아직 구현되지 않았고 현재 계획으로도 확정되지 않은 내용은 상세 설명 대신 템플릿만 남긴다.

## 권장 서술 항목

구조나 계약을 설명하는 문서에는 필요에 따라 아래 항목을 우선 포함한다.

* 목적
* 현재 범위
* 주요 책임
* 관련 코드 경로
* 입력/출력 또는 단계 경계
* 수정이 필요한 트리거

## 유지 규칙

* 코드 경로나 계약이 바뀌면 같은 작업에서 문서를 함께 갱신한다.
* 파일 추가, 삭제, 이동 시 `docs/README.md`도 같이 갱신한다.
* 현재 코드와 맞지 않는 문장, 다른 저장소에서 가져온 템플릿 표현, 사용하지 않는 계약은 제거한다.
* 미래 확장 항목이 필요하면 `상태: TBD` 또는 템플릿 필드만 두고 상세 동작은 확정 후 작성한다.
