# 개발 계획

## 목적

이 문서는 현재 저장소에서 먼저 만들어야 할 동작 가능한 최소 제품(MVP)의 개발 계획을 설명한다.

목표는 아키텍처 경계를 완벽하게 분리하는 것이 아니라, 사용자가 웹 UI에서 종목을 입력하고, 백엔드 워크플로우가 판단 결과와 주문 계획을 만들고, 그 결과를 저장한 뒤 다시 조회할 수 있는 전체 제품 흐름을 완성하는 것이다.

투자 전략, 실제 외부 데이터 API, 실브로커 주문, 고급 분석 지표는 이 문서의 현재 범위가 아니다. 해당 항목은 MVP 이후 확장 단계에서 정한다.

## MVP 목표

MVP는 아래 흐름이 끝까지 동작하는 것을 목표로 한다.

```text
사용자
-> 웹 UI에서 종목 입력
-> 백엔드 API 호출
-> MainAgent 워크플로우 실행
-> 데이터 수집 stub
-> 분석 stub
-> 리스크 검증 stub
-> 매수/매도 판단
-> 주문 계획 생성
-> 평가 로그 생성
-> 결과 저장
-> 웹 UI에서 결과와 히스토리 확인
```

현재 MVP에서는 실제 외부 API와 실주문을 호출하지 않는다.

## 핵심 설계 원칙

* 먼저 끝까지 동작하는 제품 흐름을 만든다.
* 실제 데이터, 실제 브로커, 실제 저장소 확장은 MVP 이후 단계로 둔다.
* `MainAgent`는 현재 워크플로우 순서를 유지한다.
* 주문 계획은 생성하지만 실제 주문 제출은 하지 않는다.
* 결과 저장과 조회는 MVP 제품 기능으로 포함한다.
* 구조 분리는 제품 흐름을 방해하지 않는 최소 범위에서 진행한다.
* 미확정 투자 전략과 브로커 세부 계약은 `TBD`로 남긴다.

## 1주차: 현재 앱을 실제 사용 가능한 흐름으로 정리

목표는 웹에서 종목을 입력하면 백엔드 판단 결과가 화면에 나오는 것이다.

작업:

* FastAPI `/decisions` 응답 구조 정리
* 프론트엔드에서 심볼 입력 폼 구현
* 프론트엔드에서 `/decisions` 호출
* decision, order plan, risk result를 화면에 표시
* 에러 상태와 로딩 상태 추가
* 현재 stub workflow는 유지

산출물:

* 사용자가 `AAPL, MSFT` 같은 심볼을 입력할 수 있다.
* 결과 화면에서 판단과 주문 계획을 확인할 수 있다.
* 백엔드와 프론트엔드가 연결된다.

## 2주차: 결과 저장 기능 추가

목표는 한 번 생성한 판단 결과를 다시 볼 수 있게 하는 것이다.

작업:

* `repositories/` 패키지 추가
* `DecisionLogRepository` interface 정의
* 초기 저장 구현 추가
* decision 생성 시 결과 저장
* `GET /decisions` 목록 API 추가
* `GET /decisions/{id}` 상세 API 추가

초기 저장 구현은 SQLite를 우선 검토한다. 단, MVP 속도가 더 중요하면 memory repository로 시작하고 SQLite는 다음 작업으로 넘긴다.

산출물:

* 판단 기록이 저장된다.
* 과거 판단 목록을 조회할 수 있다.
* 특정 판단 상세를 조회할 수 있다.

## 3주차: 웹 UI에 히스토리 화면 추가

목표는 사용자가 이전 판단들을 볼 수 있는 최소 운영 화면을 만드는 것이다.

작업:

* 최근 decision 목록 표시
* decision 상세 보기 구현
* 새 판단 생성 화면과 상세 화면 연결
* 결과 화면에 요청 심볼, 분석 점수, 리스크 승인 여부 표시
* 결과 화면에 매수/매도/HOLD 판단 표시
* 결과 화면에 주문 제출 가능 여부와 평가 요약 표시

산출물:

* 단발성 API demo가 아니라 제품처럼 사용할 수 있다.
* 새 판단 생성과 과거 판단 조회가 같은 UI 안에서 가능하다.

## 4주차: 아키텍처 경계 최소 분리

목표는 제품 흐름을 유지하면서 나중에 실제 데이터와 브로커를 붙일 수 있는 최소 경계를 만드는 것이다.

작업:

* `providers/` 패키지 추가
* `DataCollectionAgent`의 stub 데이터 생성 로직을 provider 경계로 이동
* `brokers/` 패키지 추가
* `OrderExecutionAgent`가 실제 주문 없이 주문 계획만 생성하는 계약 유지
* 브로커 adapter 추가를 위한 기본 경계 준비
* `api/dependencies.py`를 추가해 `MainAgent` 조립 로직 분리

산출물:

* 전체 제품 동작은 유지된다.
* 데이터 provider 교체가 가능해진다.
* 브로커 adapter 추가를 위한 자리가 생긴다.
* API 진입점과 조립 코드가 분리된다.

## 5주차: 구조 테스트와 API 테스트 보강

목표는 MVP 흐름이 깨지지 않게 최소 회귀 테스트를 만드는 것이다.

작업:

* `POST /decisions` 테스트 추가
* `GET /decisions` 테스트 추가
* `GET /decisions/{id}` 테스트 추가
* `MainAgent` workflow 테스트 유지
* 리스크 미승인 시 주문 제출 불가 테스트 유지
* repository 저장과 조회 테스트 추가

산출물:

* 핵심 제품 흐름 테스트가 생긴다.
* API 계약 테스트가 생긴다.
* 저장소 테스트가 생긴다.

## 6주차: UI 정리와 사용자 경험 보강

목표는 돌아가는 수준에서 실제로 쓸 수 있는 수준으로 UI를 정리하는 것이다.

작업:

* 빈 입력 방어
* 심볼 정규화 표시
* API 실패 메시지 표시
* 판단 결과 카드 또는 테이블 정리
* 리스크 거절 사유 표시
* 주문 계획이 실제 주문이 아님을 명확히 표시
* 모바일과 데스크톱 레이아웃 확인

산출물:

* 최소 제품 UI가 완성된다.
* 사용자가 전체 흐름을 이해할 수 있다.
* 실주문 오해를 줄인다.

## 우선순위

MVP의 작업 우선순위는 아래 순서로 본다.

```text
1. 웹에서 판단 생성
2. 백엔드 workflow 결과 반환
3. 결과 저장
4. 히스토리 조회
5. 최소 구조 분리
6. 테스트
7. 문서
8. 품질 검증
```

## MVP 이후 확장 후보

아래 항목은 MVP 이후에 별도 계획으로 확정한다.

* 실제 시장 데이터 provider
* 고급 analyzer 구조
* 리스크 policy 세분화
* 한국투자증권 Open API broker adapter
* SQLite 또는 PostgreSQL 기반 영속 저장 고도화
* 백테스트
* 모의투자
* 제한적 실거래

## 수정 트리거

아래 상황이 생기면 이 문서를 갱신한다.

* MVP 범위가 바뀜
* API endpoint 계약이 바뀜
* UI의 핵심 제품 흐름이 바뀜
* 저장 방식이 바뀜
* 실제 데이터, 브로커, 백테스트가 MVP 범위로 들어옴
* 주차별 작업 순서나 검증 기준이 바뀜
