# TESTS GUIDE

## 목적

테스트는 현재 저장소가 보장해야 하는 동작 계약을 검증하기 위한 수단이다.
구현 세부사항보다 사용자 관점의 동작과 시스템 계약을 우선 확인한다.

## 기본 원칙

* 현재 코드와 문서에 남아 있는 계약만 검증한다.
* public behavior와 business contract를 우선 테스트한다.
* repo-relative entrypoint 기준으로 테스트를 구성한다.
* 구현 세부사항에 과하게 결합된 테스트는 피한다.

## 이 저장소에서 우선 검증할 것

* 오케스트레이터가 에이전트 단계를 올바른 순서로 조합하는지
* 스키마 기반 입출력이 현재 계약을 만족하는지
* 주문 실행 단계가 현재 최소 구현 범위 안에서 기대된 결과를 반환하는지
* 보고서 단계의 리스크 검증이 매수/매도 판단 전에 수행되는 현재 계약이 유지되는지

## 테스트를 추가해야 하는 경우

* 새로운 공개 동작이 생김
* 기존 계약이 수정됨
* 회귀 가능성이 있는 버그를 수정함
* 사용자에게 보이는 응답 형식이나 흐름이 바뀜

## 테스트를 추가하지 않아도 되는 경우

* 문서만 수정한 경우
* 로컬 개발 편의용 스크립트나 scratchpad만 바뀐 경우
* 공개 계약과 무관한 사소한 내부 정리만 있는 경우

## 실행 원칙

* 기본 테스트 명령은 `UV_CACHE_DIR=/tmp/uv-cache uv run pytest` 또는 `UV_CACHE_DIR=/tmp/uv-cache uv run poe test`를 사용한다.
* 테스트를 실행하지 못하면 생략 사유와 남은 위험을 결과에 남긴다.
* 구현이 안정된 뒤 formatting과 linting을 진행한다.

## 현재 테스트 초점

현재 `tests/test_orchestrator.py`는 아래 최소 계약을 검증한다.

* `MainAgent`가 최소 워크플로우를 끝까지 실행하는지
* 보고서 단계의 리스크 미승인 시 주문 계획이 제출 불가 상태로 남는지
